## Bulgular

### HIGH-1 — `Rollback` publish-safety check’i bypass ediyor

**Dosyalar:**
- `internal/game/admin/content_service.go:289-358` (`PublishDraft`)
- `internal/game/admin/content_service.go:361-430` (`Rollback`)

**Problem:** `PublishDraft` current snapshot’i yükledikten sonra runtime apply plan çıkarıyor ve hemen `validatePublishSafety(ctx, current.Snapshot, snapshot)` çağırıyor. Aynı sınıftaki `Rollback` ise current snapshot + target snapshot ile `RuntimeApplyPlan` hesaplıyor ama **publish safety validation çağırmadan** `PublishContentSnapshot` yapıyor.

Sonuç: Aktif craft/production/equipped-module referansını değiştiren normal publish bloklanırken, aynı değişikliği eski bir versiyona rollback ile uygulamak mümkün.

**Neden önemli:** P14 done criteria “active references require quiescence” diyor. Rollback de canlı current content’i değiştiren bir publish operation; aynı safety gate’ten geçmeli. Aksi halde admin, aktif loadout’taki module definition’ını rollback ile değiştirebilir.

**Öneri:** `Rollback` içinde `result.RuntimeApplyPlan = ...` satırından hemen sonra aynı safety call’u ekle:

```go
if err := service.validatePublishSafety(ctx, current.Snapshot, snapshot); err != nil {
    return content.PublishDraftResult{}, err
}
```

**Test önerisi:**
- Current snapshot: `laser_alpha_t1` v2.
- Target rollback snapshot: `laser_alpha_t1` v1.
- Runtime/content service active equipped reader `laser_alpha_t1` döndürsün.
- `Rollback` `ErrContentPublishActiveEquippedModule` ile fail etmeli ve yeni content version/audit row oluşmamalı.

### HIGH-2 — Safe reload “applied” raporluyor ama server read-model’leri eski kalıyor

**Dosyalar:**
- `internal/game/content/content_apply.go:24-31`
- `internal/game/server/runtime_content_apply.go:34-49`
- `internal/game/server/shop_handlers.go:98-110`, `140-147`, `203-224`

**Problem:** `safeReloadContentTypes` item/module/shop_product değişikliklerini safe kabul ediyor. Fakat `Runtime.applyPublishedContent` sadece `contentCatalogProjection` ve `contentCatalogVersion` swap ediyor. Boot’ta kurulan authoritative/read model alanları eski kalıyor:

- `runtime.Content` → `shop.catalog` ve `shop.buy_product` burada okuyor.
- `runtime.ModuleCatalog` → loadout/stat/combat/repair path’leri burada okuyor.
- `runtime.itemCatalog` → inventory/shop grant validation burada okuyor.

Bu yüzden admin response `runtime_applied=true` diyebilir, `content.catalog` yeni projection gösterebilir; ama `shop.catalog`, `shop.buy_product`, loadout/stat hesapları eski content ile devam eder.

**Somut örnekler:**

1. CMS’te `shop_product` price/availability/rank değişti ve publish safe-reload döndü.
   - `content.catalog` yeni projection’ı gösterebilir.
   - `shop.catalog` hâlâ `runtime.Content.SortedShopProducts()` ile eski fiyat/availability döndürür.
   - `shop.buy_product` hâlâ eski fiyat/rank üzerinden debit/grant yapar.

2. CMS’te module stat modifier/display değişti ve safe-reload döndü.
   - `content.catalog` yeni module stat projection gösterebilir.
   - server-side `ModuleCatalog` eski kaldığı için stat service/combat eski değerleri kullanır.
   - Bu, “client only intent, server truth” kuralına ters algı yaratır: client’a yeni stat gösterilir ama server eski truth ile oynatır.

**Neden önemli:** P14 hedefi “publish canlı runtime’a yansır veya dürüstçe pending_restart raporlar.” Burada third state var: projection yansıyor, gameplay/read-model yansımıyor, ama response `runtime_applied=true` diyebiliyor.

**Öneri seçenekleri:**

- Kısa vadeli güvenli fix: `ContentTypeModule` ve `ContentTypeShopProduct` için `restart_required` dön; yalnız display-only item projection gibi gerçekten presentation-only alanları safe bırak.
- Daha doğru fix: safe-reload sırasında `runtime.Content`, `runtime.ModuleCatalog`, `runtime.itemCatalog` ve ilgili derived read-model’leri atomik şekilde swap et; handler’lar aynı version’ı görsün.
- Field-level classification: row’un sadece `display_json` değiştiği durumlar safe; `data_json` gameplay/price/stat/availability değişiklikleri restart-required.

**Test önerisi:**
- Safe-reload shop product price değişikliğinden sonra ya:
  - `shop.catalog` ve `shop.buy_product` yeni fiyatı kullanmalı, **ya da**
  - publish response `runtime_applied=false`, `pending_restart=true` dönmeli.
- Module stat değişikliği için `content.catalog` projection ile server stat result aynı version/değerde olmalı; değilse pending restart.

### HIGH-3 — Ship shop purchase wallet debit + hangar grant tek transaction değil

**Dosyalar:**
- `internal/game/content/shop.go` — non-starter ship artık available.
- `internal/game/server/shop_handlers.go:163-174`

**Problem:** `handleShopBuyProduct` önce wallet debit yapıyor, sonra `grantShopProductLocked` çağırıyor. Grant path hata dönerse wallet debit committed kalıyor, hangar/item grant yok, `runtime.shopPurchases[referenceKey]` de yazılmıyor.

Bu risk zaten item/module shop path’inde de vardı; fakat `912bc65` ile ilk gerçek ship purchase path’i açıldığı için P07 lane-C artık bu non-transactional akışa dayanıyor.

**Neden önemli:** AGENTS ekonomi kuralı açık: value mutation `lock -> validate -> mutate -> ledger/event -> commit -> broadcast` transaction-style olmalı. Wallet debit + hangar unlock iki ayrı authoritative state mutation; arada hata/retry oyuncudan kredi alıp gemiyi vermeyebilir.

**Öneri:**

- Shop purchase için domain service/repository transaction seam’i ekle: wallet debit + inventory/hangar grant + ledger/outbox + idempotency aynı transaction boundary’de olsun.
- En azından in-memory fallback’te failure-injection test’i ile debit sonrası grant fail durumunu yakala; durable store’a geçerken gerçek transaction şart.
- `shop_purchase:<request_id>` idempotency kaydı durable olmalı; process-local `runtime.shopPurchases` restart sonrası retry güvenliği vermez.

**Test önerisi:**
- Fake `HangarStore`/`HangarService` grant sırasında hata döndürsün.
- `shop.buy_product` fail olduğunda wallet balance unchanged kalmalı ve ledger row oluşmamalı.
- Retry aynı request_id ile double debit yapmamalı.
